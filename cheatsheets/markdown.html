Markdown Cheat Sheet

Syntax reference for CommonMark and GitHub Flavored Markdown — from structure and emphasis to tables, escaping, and common gotchas.

Section 01

Headings

Headings organise content into a hierarchy. Markdown supports two styles — ATX (hash prefix) is almost always preferred.

ATX style

Markdown HTML output Notes
# Heading 1<h1>Top-level — use once per page
## Heading 2<h2>Primary section
### Heading 3<h3>Sub-section
#### Heading 4<h4>Rarely needed
##### Heading 5<h5>Very rarely needed
###### Heading 6<h6>Almost never needed

Setext style (H1 / H2 only)

Heading 1
=========

Heading 2
---------
Pro Tip
Always use ATX style. Setext is harder to read in source, only supports two levels, and the --- underline is easily confused with a horizontal rule.
Watch Out
Never skip heading levels for visual effect — jumping from ## H2 straight to #### H4 breaks accessibility and document structure. Use CSS if you want different sizes.
Section 02

Paragraphs & Line Breaks

Markdown's most-misunderstood rule: a single newline is invisible. You need a blank line to start a new paragraph.

What you write What you get Notes
Two blocks separated by a blank line Two separate paragraphs Standard paragraph break
Two lines with no blank line One paragraph joined with a space Single newlines are treated as spaces
Line ending with two trailing spaces Hard line break inside the paragraph Invisible and easily stripped by editors
<br> at end of line Hard line break inside the paragraph Explicit and reliable — prefer this
Wrong — one newline
First line
Second line

Renders as a single paragraph
Right — blank line between
First paragraph

Second paragraph

Remember
Prefer <br> over trailing-space line breaks. Two trailing spaces are invisible, look like nothing, and are silently removed by most linters and editors on save.
Section 03

Emphasis

Bold, italic, strikethrough, and combinations. Both * and _ work for most cases — with one important caveat around mid-word use.

Markdown Renders as Notes
*italic* or _italic_italicSingle delimiter
**bold** or __bold__boldDouble delimiter
***bold italic***bold italicTriple delimiter; also ___…___
~~strikethrough~~strikethroughGFM only — not standard CommonMark
`inline code`inline codeBacktick — monospace, no parsing inside
Watch Out
Underscores do not work mid-word: some_var_name will trigger unwanted italics in many parsers. Use asterisks for intra-word emphasis (some*var*name), or escape the underscores (some\_var\_name).
Convention
Pick one delimiter style and use it everywhere. Most style guides prefer **bold** over __bold__ and *italic* over _italic_ — asterisks have fewer parser edge cases.
Section 04

Blockquotes

Blockquotes use a > prefix. They can nest, span multiple paragraphs, and contain other Markdown formatting.

Markdown Notes
> Single-line quote Basic blockquote
> Line 1
> Line 2		 Multi-line — repeat > on each line
> Para 1
>
> Para 2		 Multi-paragraph — use a bare > as the blank line
> Outer
>> Nested		 Nested blockquote — add another >
> Quote with **bold** and `code` Markdown formatting works inside quotes 
> "Not all those who wander are lost."
>
> The quote continues in a second paragraph.
>> A nested quote inside the above.
Pro Tip
Many platforms (GitHub, Obsidian, Notion) support alert-style blockquotes: > [!NOTE], > [!WARNING], > [!TIP]. They render with coloured callout styling. Check your platform's docs for the exact syntax.
Section 05

Lists

Markdown supports unordered, ordered, nested, and task lists. Indentation controls nesting — consistency matters here.

Unordered lists

- Item one        ← dash (recommended)
* Item two        ← asterisk (also valid)
+ Item three      ← plus (also valid)
Remember
All three markers produce identical output, but mixing them within a single list creates separate lists in most parsers. Pick - and stick with it.

Ordered lists

1. First item
2. Second item
3. Third item
Lazy Numbering
You can write every item as 1. — most parsers auto-increment. This makes reordering easier, but the source is harder to scan at a glance.

Nested lists — indent by 2 or 4 spaces

- Top-level item
  - Nested item (2-space indent)
    - Deeper item (4-space indent)
- Another top-level item

Task lists (GitHub Flavored Markdown)

- [x] Completed task
- [ ] Incomplete task
- [ ] Another item
Pro Tip
Task lists render as interactive checkboxes on GitHub issues and pull requests. They're a clean way to track sub-tasks without a separate project board.
Watch Out
A blank line between list items creates a loose list — each item gets wrapped in a <p> tag, adding vertical spacing. Remove the blank lines for a tight, compact list.
Section 06

Code

Two formats: inline backticks for short snippets within a sentence, and fenced blocks for multi-line code with optional syntax highlighting.

Inline code

Use the `print()` function to output text.

If your snippet contains a backtick, wrap it in double backticks: `` `tick` ``

Fenced code blocks

```python
def greet(name):
    return f"Hello, {name}!"
```
Language tag What gets highlighted
pythonPython

(no tag)Plain monospace, no highlighting

Indented code block (legacy style)

    Indent by 4 spaces or 1 tab to create a code block.
    Prefer fenced blocks — they support language tags and
    are harder to accidentally trigger.
Pro Tip
Always specify a language tag — even for plain text. It signals intent to readers and prevents tools from running auto-detection, which can produce false positives on short snippets.
Section 07

Links & Images

Links and images share the same syntax — images just add a ! prefix. Both support inline and reference styles.

Markdown Notes
[Link text](https://example.com) Inline link
[Link text](https://example.com "Tooltip") Inline link with hover tooltip
[Link text][ref-id]
[ref-id]: https://example.com		 Reference-style — definition can go anywhere in the file
<https://example.com> Auto-link — URL becomes its own clickable text
https://example.com Bare auto-link (GFM only)
![Alt text](image.png) Inline image
![Alt text](image.png "Caption") Image with title tooltip
[![Alt](img.png)](https://example.com) Clickable image — image wrapped in a link
Pro Tip
Use reference-style links when a URL appears multiple times or when long URLs clutter the prose. Group all definitions at the bottom of the file for easy maintenance.
Watch Out
Always write descriptive alt text. ![photo.png](photo.png) is meaningless to screen readers. Describe what the image communicates, not just what it depicts: ![Bar chart showing 40% growth in Q3](chart.png).
Section 08

Tables

Tables are a GitHub Flavored Markdown (GFM) extension — not part of standard CommonMark. Pipes separate columns; a separator row with dashes defines the header.

| Name    | Role      | Active |
| ------- | --------- | ------ |
| Alice   | Engineer  | Yes    |
| Bob     | Designer  | No     |

Column alignment

| Left    | Centre    | Right  |
| :------ | :-------: | -----: |
| text    | text      | text   |
Separator syntax Alignment
--- or :---Left (default)
:---:Centre
---:Right
Limitations
Tables cannot contain multi-line cells, and complex formatting inside cells is limited. For anything more structured, fall back to raw HTML.
Remember
Outer pipes (at the start and end of each row) are optional but strongly recommended for readability. The dashes in the separator row don't need to match the column width — | --- | is enough.
Section 09

Horizontal Rules

A horizontal rule renders a thematic break — a visible dividing line between sections of content.

Markdown Notes
---Three or more dashes (most common)
***Three or more asterisks
___Three or more underscores
- - - or * * *Spaces between characters are also valid
Watch Out
--- placed directly under a line of text is interpreted as a Setext H2 heading, not a horizontal rule. Always leave a blank line above --- if you want a rule.
Danger — creates an H2 heading

---
Correct — creates a horizontal rule

Section 11

Common Mistakes

The errors that trip up even experienced Markdown writers — most come down to whitespace and parser behaviour.

1
One newline ≠ new paragraph A single line break is absorbed as a space. You need a blank line between blocks to create a new paragraph.
2
Underscore mid-word some_var_name can trigger unwanted italics. Use asterisks for intra-word emphasis, or escape: some\_var\_name.
3
--- after text = heading Dashes directly under a line of text create an H2, not a rule. Always leave a blank line above --- if you want a horizontal rule.
4
Missing space after # #Heading is not a heading in most parsers — they require # Heading with a space. Without it you get literal hash characters.
5
Mixing list markers Using -, *, and + in the same list restarts the list in many parsers. Pick one marker and keep it consistent.
6
Loose vs tight lists Blank lines between list items wrap each item in <p> tags, adding vertical space. Omit blank lines for a compact list.
7
Trailing-space line breaks Two trailing spaces produce a <br>, but they're invisible and silently stripped by most editors on save. Use explicit <br> instead.
8
Tables aren't CommonMark Tables are a GFM extension. They won't render in standard CommonMark parsers, Pandoc defaults, or many static site generators.
9
Empty image alt text ![](image.png) leaves screen readers silent. Always write alt text that conveys the image's meaning, not just its filename.
10
Formatting inside backticks Markdown is not parsed inside inline code: `**not bold**` renders literally. Use HTML if you genuinely need formatted code text.
Pro Tip
When Markdown doesn't render as expected, check three things: (1) missing blank lines, (2) inconsistent indentation, and (3) which parser you're using — CommonMark, GFM, and MultiMarkdown have real behavioural differences.
