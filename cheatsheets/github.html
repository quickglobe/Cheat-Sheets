GitHub Cheat Sheet

Git and GitHub from the ground up — commits, branches, pull requests, and the decisions you make every time you click Merge.

Section 01

The Big Picture

Git and GitHub are two different things that work together. Understanding the distinction — and the four zones your code moves through — makes everything else click.

Git vs GitHub

Git is version-control software that runs on your computer. It tracks changes, manages branches, and maintains your project's full history — all locally, with no internet required. GitHub is a website that hosts Git repositories in the cloud so teams can share code, review changes, and merge work together.

When Claude Code runs git commit, that's a local operation. Nothing reaches GitHub until something explicitly pushes.

The four zones

Every change you make moves through four places before it's on GitHub:

ZoneWhat lives hereHow to move forward
Working directory Files you've edited but haven't told Git about yet git add <file>
Staging area Changes you've flagged to include in the next commit git commit -m "..."
Local history Committed snapshots — safe, permanent, still private git push
GitHub (remote) Shared history visible to everyone with repo access Open a pull request to merge into main

Commits are snapshots, not diffs

Each commit is a complete snapshot of every tracked file at a point in time — not a list of line-by-line changes. Git figures out what changed when you ask it to; that's just a display convenience. The upside: any commit can be restored exactly, and history never becomes corrupted by accumulated patches.

Remember
Claude Code operates entirely in your local repo — it stages files, writes commits, and pushes branches. The only thing you need to do on GitHub itself is review and merge pull requests.
When to Use GitHub
You only need to visit GitHub when you're sharing work (reviewing a PR someone else opened) or finishing work (merging a PR into main). Day-to-day coding happens locally.
Section 02

Branches

A branch is a moveable label that points to a commit. Creating one is free and instant — it's just a pointer, not a copy of your files.

Why branches exist

Branches let you isolate work so that main always stays in a working state. You develop a feature on its own branch, then merge it in when it's ready. If the feature goes sideways, you delete the branch — main is untouched. Multiple branches can exist at once, letting several features progress in parallel.

main (or master in older repos) is the branch that represents your finished, deployable code. Feature branches are short-lived branches where work actually happens. Claude Code automatically creates feature branches named like claude/descriptive-name-ABC123 — you'll see these as the source branch when it opens a pull request.

CommandWhat it does
git branch List all local branches; current branch is marked with *
git branch -a List local and remote branches
git switch -c new-branch Create a new branch and switch to it immediately
git switch branch-name Switch to an existing branch
git checkout -b new-branch Older equivalent of switch -c — you'll see both in docs
git branch -d branch-name Delete a branch (safe — won't delete if unmerged)
git branch -D branch-name Force-delete a branch, even if unmerged
Pro Tip
Never commit directly to main. Even when working alone, the habit of using feature branches means you always have a clean main to fall back on, and your history stays readable.
Section 03

Staging & Committing

A commit is a deliberate checkpoint — not an autosave. The staging area is what lets you choose exactly which changes go into each checkpoint.

Why staging exists

Imagine you've edited three files: a new feature, a bug fix, and a typo correction. You could lump them into one commit, but that makes history noisy and harder to understand later. The staging area lets you add the bug fix alone, commit it with a focused message, then stage and commit the other changes separately — even though they all changed at the same time.

CommandWhat it does
git status Show which files are modified, staged, or untracked
git add filename Stage a specific file
git add . Stage every changed file in the current directory
git add -p Interactively stage chunks of a file (advanced)
git diff Show unstaged changes
git diff --staged Show staged changes (what will go into the next commit)
git commit -m "message" Commit staged changes with an inline message
git commit Open your editor to write a longer commit message
git log --oneline Show recent commits as a compact single-line list
git show abc1234 Show the full diff introduced by a specific commit

Writing a good commit message

The subject line should explain why the change was made, not just what changed — the diff already shows the what. A reader should be able to understand the intent without opening the files.

Vague
fix bug

update styles

changes
Useful
fix login redirect loop when session expires

increase nav button contrast for accessibility

remove deprecated payment API calls
When Using Claude Code
Claude Code stages and commits for you. Run git log --oneline afterwards to see what it did — it's a useful habit for staying aware of your project's history.
Watch Out
Don't use git commit --amend on a commit that has already been pushed to GitHub. Amending rewrites history, which conflicts with what's on the remote and forces a disruptive push. Only amend commits that are still local.
Section 04

Pushing & Pulling

Pushing uploads your local commits to GitHub; pulling downloads the latest from GitHub into your local repo. Neither happens automatically.

CommandWhat it does
git push Upload local commits to the tracked remote branch
git push -u origin branch-name Push and set up tracking — so future git push knows where to go
git push --force-with-lease Force push safely — fails if someone else has pushed since you last fetched
git fetch Download remote changes without touching your working files
git pull Fetch and then merge remote changes into your current branch
git pull --rebase Fetch and replay your local commits on top (avoids merge commits)
git remote -v Show the remote URLs your repo is connected to

What -u origin branch-name means

origin is the default nickname Git gives to the remote repo (usually your GitHub repo). The -u flag tells Git to remember this remote branch as the default target for future pushes and pulls from this branch. After the first push with -u, you can just type git push — Git already knows where to send it. Claude Code always uses this pattern when it first pushes a new branch.

fetch vs pull

git fetch is safe — it downloads remote commits into your local Git database without changing any of your files. git pull does a fetch then immediately merges the result into your current branch, which can introduce a merge commit or conflict. When you're unsure what changed remotely, fetch first, inspect with git log origin/main, then decide whether to merge.

Remember
git status is your best friend. Run it before and after any Git operation to understand exactly what state your repo is in — what's staged, what's not, and whether you're ahead of or behind the remote.
Section 05

Pull Requests

A pull request is a proposal to merge one branch into another, with a built-in place for discussion, review, and automated checks — before anything touches main.

Anatomy of a pull request

PartWhat it is
Base branch The branch you want to merge into — almost always main
Compare branch The branch with your changes — what Claude Code pushed
Description Explains what changed, why, and how to test it
Reviewers People asked to read and approve the changes
CI checks Automated tests and linters that run on every push — shown as green ticks or red crosses
Merge button Appears once checks pass and approvals are in place — clicking it applies the merge strategy you choose

Draft vs ready for review

A draft PR signals that work is in progress and not ready for review yet — CI still runs, but reviewers know not to dig in. A ready-for-review PR is requesting feedback. Claude Code always opens PRs as ready for review.

Reviewing a PR

On the Files changed tab you'll see every line that was added (green) or removed (red). You can click on any line to leave an inline comment. When you're done, choose one of three review outcomes: Approve (looks good), Request changes (needs work before merging), or Comment (observations without a verdict).

Pro Tip
Read the PR description before diving into the code — Claude Code writes it to explain what changed and why. It saves time and gives you context that makes the diff much easier to follow.
Remember
Wait for CI checks to go green before merging. A red check means a test failed or the linter found a problem — merging anyway brings a broken state into main.
Section 06

Merge Strategies

When you click the merge button on GitHub, you choose how the branch's commits are applied to main. The three options produce very different histories.

StrategyWhat it creates on mainBest for
Merge commit All original commits, plus a new merge commit tying the branches together Long-running branches where you want to preserve every commit for audit purposes
Squash and merge All branch commits collapsed into one single new commit on main Feature branches — one PR becomes one clean commit; the branch is disposable
Rebase and merge Each commit from the branch replayed on top of main, one by one, no merge commit When you want linear history and each commit on the branch is already clean and meaningful

What each strategy looks like in git log

Suppose a feature branch has three commits: Create auth module, Add login form, Fix validation.

# Merge commit — full branch history preserved, plus a merge node
*   a1b2c3d  Merge pull request #42: Add user login
|\
| * f0e1d2c  Fix validation
| * 9a8b7c6  Add login form
| * 3d4e5f6  Create auth module
|/
*   e5f6a7b  Previous commit on main


# Squash and merge — all three commits become one
*   a1b2c3d  Add user login (#42)
*   e5f6a7b  Previous commit on main


# Rebase and merge — commits land individually, no merge node
*   f0e1d2c  Fix validation
*   9a8b7c6  Add login form
*   3d4e5f6  Create auth module
*   e5f6a7b  Previous commit on main
Recommendation
Squash and merge is the right choice for most feature branches. Your branch's work-in-progress commits (typo fixes, "oops" commits) disappear, and main's history becomes a clean list of completed features — one entry per PR.
Watch Out
Never rebase a branch that someone else is working on. Rebase rewrites commit hashes, so anyone who has based work on the old commits will have a broken history that's painful to reconcile.
Section 07

Merge Conflicts

A conflict happens when two branches each changed the same lines of the same file — Git can't know which version to keep, so it stops and asks you to decide.

What causes a conflict

Most merges are automatic. Conflicts only occur when both the source branch and the target branch modified the exact same region of a file since they diverged. Unrelated changes to different files, or even different parts of the same file, merge cleanly without any intervention.

Reading conflict markers

When Git encounters a conflict, it edits the file in place and wraps the competing versions in markers:

<<<<<<< HEAD
const timeout = 5000;
=======
const timeout = 10000;
>>>>>>> feature/increase-timeout

Everything between <<<<<<< HEAD and ======= is what your current branch has. Everything between ======= and >>>>>>> is what the incoming branch has. Your job is to edit the file so it contains only the final version you want — no markers left behind.

Resolving a conflict

StepCommand / action
1. See which files have conflicts git status — conflicted files are listed under "both modified"
2. Open the file and edit it Delete the markers and both conflicting versions, leaving only what you want
3. Mark the conflict resolved git add <filename>
4. Complete the merge git commit — Git pre-fills the merge commit message

VS Code conflict UI

If you open a conflicted file in VS Code, it highlights the conflict regions and shows clickable actions above each one: Accept Current Change (keep HEAD), Accept Incoming Change (keep the branch), Accept Both Changes (keep both, stacked), or Compare Changes (diff view). For straightforward conflicts this is much faster than editing the markers manually.

Pro Tip
Most conflicts look scarier than they are. The two versions are usually just different values or slightly different function calls — you read them, pick the right one (or combine them), delete the markers, and you're done.
Using Claude Code
If you hit a conflict you're unsure about, you can ask Claude Code to resolve it. Describe which version or intent should win, and it will edit the files, remove the markers, and commit the resolution.
Watch Out
Never commit a file that still contains conflict markers (<<<<<<<, =======, >>>>>>>). They are literal text in the file — your app will break, your tests will fail, and it'll be obvious to anyone reading the code that the conflict was never properly resolved.
Section 08

GitHub CLI Quick Reference

The gh CLI lets you interact with GitHub from the terminal — check PR status, create PRs, and merge branches without leaving the command line.

One-time setup

Run gh auth login once and follow the prompts. After that, all gh commands authenticate automatically.

CommandWhat it does
gh pr list Show all open pull requests in the repo
gh pr view Show the PR for the current branch in the terminal
gh pr view --web Open the current branch's PR in your browser
gh pr create Create a PR for the current branch interactively
gh pr create --title "..." --body "..." Create a PR with title and description inline (non-interactive)
gh pr merge Merge the current branch's PR — prompts you to choose the merge strategy
gh pr merge --squash Merge with squash strategy directly, no prompt
gh pr checkout 42 Check out PR #42 locally so you can inspect or run the code
gh pr status Show the status of PRs relevant to you (created, assigned, review requested)
gh issue list List open issues in the repo
gh issue view 15 Read issue #15 in the terminal
gh repo clone owner/repo Clone a GitHub repository by its owner/name shorthand
gh repo view --web Open the current repo on GitHub in your browser
Pro Tip
Run gh pr view from inside any branch Claude Code created and you'll immediately see the PR it opened — including CI status, reviewers, and a direct link. No need to search GitHub.
Remember
gh pr merge --squash from the terminal does exactly the same thing as clicking "Squash and merge" on GitHub. If you're already in the terminal, there's no reason to switch to the browser just to merge.
