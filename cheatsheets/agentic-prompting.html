Agentic Prompting

From chat interface to autonomous agents — for intermediate practitioners moving into agentic and API environments.

Section 01

The Mindset Shift: Chat vs Agentic

In a chat interface, you prompt once and read one response. In an agentic environment, the model plans, acts, observes results, and loops — often without you watching. This changes almost everything about how you write prompts.

1
Statelessness by default Each API call is memoryless. Context — history, file contents, prior results — must be explicitly included every time. Nothing is "remembered" unless you pass it in.
2
Multi-step, not single-shot Agents execute chains of actions across an indeterminate number of steps. A prompt is a standing instruction, not a one-off message. Errors compound — a wrong early step corrupts everything downstream.
3
The model directs itself Unlike a workflow with hard-coded step A → B → C, an agent dynamically decides what to do next. Your prompt defines the goal and constraints; the model chooses the path.
4
Actions have real consequences An agent can write files, call APIs, push code, send emails. Mistakes aren't just wrong words — they may be irreversible. Guardrails matter far more than in chat.
5
Context is a finite resource The context window fills up as the agent works. Bloated context degrades performance (called "context rot"). You must actively manage what goes in and what gets trimmed.
6
Tools are first-class citizens In chat you use tools opportunistically. In agentic systems, tool definitions, naming, and documentation are as important as the prompt itself — the model reads them as instructions.

At a glance: Chat vs Agentic

DimensionChat InterfaceAgentic / API
Prompt scopeOne-off messageStanding system instruction
MemoryMaintained by the UIYou manage explicitly
Error recoveryYou correct manuallyAgent re-plans (or fails silently)
Iteration speedSeconds per loopMinutes to hours per run
Side effectsText onlyFile writes, API calls, code execution
Context growthUI handles pruningYour responsibility
Tool useOptional, opportunisticCore, carefully designed
From your existing skills
The techniques you already know — few-shot examples, chain-of-thought, role assignment, output scaffolding — all still apply in agentic contexts. This sheet covers the new layer on top: system design, context management, and tool architecture.
Section 02

Writing System Prompts for Agents

A system prompt in an agentic context is not a polite opener — it is the agent's operating manual. It runs at every turn, consumes tokens every time, and sets the boundaries of everything the agent will do.

The "right altitude" principle

Anthropic describes an optimal system prompt as sitting in a Goldilocks zone between two failure modes:

Too prescriptive
Hardcoded if-else logic in the prompt. Brittle — breaks on any case you didn't anticipate. Hard to maintain. Leaves no room for the model's reasoning.
Right altitude
Clear goal + strong heuristics + concrete examples. Specific enough to guide behaviour; flexible enough to handle novel situations. The model fills the gaps with good judgement.
Also too vague
"Help the user with their task." A system prompt so general that the model has no constraints or concrete signals is equally dangerous — especially when the agent can take real-world actions.

Recommended structure

Use XML tags or Markdown headers to separate concerns. The model parses structure, not just words.

<background_information> You are a coding agent working inside a GitHub repository. The repo is a static site built with plain HTML/CSS. </background_information> <instructions> Your job is to implement feature requests from the user. Always read the relevant file before editing it. Make the smallest change that achieves the goal. </instructions> ## Tool guidance Use `read_file` before any edit. Use `write_file` only after confirming the plan. Never delete files without asking first. ## Output description After each action, summarise what you did in one sentence.

System prompt checklist

✓
State the agent's role and environment What is it? Where does it operate? What can it see? Don't assume shared context — it has none.
✓
Define the goal, not the procedure Tell the agent what success looks like. Let it choose the steps. Only prescribe steps when a specific order is genuinely necessary.
✓
Explicit tool guidance Name which tools to prefer, when to use them, and when not to. Agents won't guess your conventions.
✓
Output format requirements What should the agent produce or report after each step? A one-line summary? A JSON object? Specify it.
✓
Hard stops and escalation rules "If you are about to delete a file, stop and ask the user." Irreversible actions need explicit checkpoints.
✓
Start minimal, expand on failure Test the shortest viable system prompt first. Add instructions only when you observe specific failure modes — not preemptively.
Pro Tip
Use capitalization or emphasis sparingly for truly critical rules: "ALWAYS read a file before editing it" or "NEVER delete without confirmation." Real-world agent prompts like Cline and Bolt use "ULTRA IMPORTANT" for their hardest constraints. Use it too liberally and it loses meaning.
Section 03

Context Engineering

Context engineering is Anthropic's term for the natural evolution of prompt engineering. Where prompt engineering asks "how should I word this?", context engineering asks "what is the optimal set of tokens to give the model at this moment?" The distinction matters because agents accumulate context over time — and too much context degrades performance.

Context rot
Research shows that as the number of tokens in the context window grows, models' ability to accurately use information from earlier in that context decreases — a phenomenon called context rot. Context is a finite resource with diminishing returns; treat it like working memory, not a filing cabinet.

What lives in context?

Context componentBest practiceCommon mistake
System prompt Clear, minimal, structured with XML or Markdown headers. Start short, add only on observed failures. Dumping every edge case into it upfront; creates noise and bloat.
Tool definitions Crisp names, unambiguous descriptions, carefully typed parameters. No overlapping tools. Too many tools; ambiguous which to use; parameter names that don't explain themselves.
Few-shot examples A small set of diverse, canonical examples that cover the expected behaviour. Quality over quantity. Listing every edge case you can think of; balloons context without proportional benefit.
Message history Summarise or prune older turns. Keep recent, high-signal turns. Use prompt caching for static parts. Sending the full unedited history for every API call; context fills up fast.
Retrieved data (RAG) Load data just-in-time via tools rather than pre-loading everything upfront. Injecting entire files or databases into context when only a fragment is needed.

Just-in-time context loading

Rather than pre-loading all potentially relevant data into the initial context, design agents to fetch data on demand using tools — file reads, database queries, web fetches. This mirrors human cognition: we don't memorise entire codebases; we open the file we need.

Tool: read_file(path) → Agent reads only the specific file it needs, when it needs it. Tool: search_codebase(query) → Agent finds and loads only the relevant snippet. Result: context stays lean throughout the run.
Pro Tip
Prompt caching (available via Anthropic's API) lets you cache static context — like a long system prompt or reference document — so the model doesn't reprocess it on every turn. This cuts both latency and cost significantly for long-running agents.
Section 04

Designing Tools Well

Tools are the agent's interface with the real world. Poor tool design is one of the most common causes of agent failure — not because the model is incapable, but because it can't use tools it doesn't understand. Think of tool documentation the way you'd think of API documentation for a developer colleague.

What is MCP?
The Model Context Protocol (MCP) is an open standard introduced by Anthropic for connecting agents to external tools and data sources. It defines a consistent way for agents to discover and call tools — think of it as a universal adapter. Claude Desktop, Claude Code, and many third-party agents support MCP out of the box.

Anatomy of a well-designed tool

Tool name: read_file Description: Read the full contents of a file at the given absolute path. Use this before editing any file. Returns the file content as a string. Do NOT use for binary files. Parameters: path (string, required) Absolute path to the file, e.g. /home/user/project/index.html Returns: String content of the file, or an error message if not found.
1
Names should be verbs Use read_file, not file. Use search_web, not web. The model picks tools by name — make the action obvious.
2
Descriptions tell when to use it Include "Use this when…" and "Do NOT use for…" in the description. The model decides which tool to call based on this text alone.
3
Parameters should be self-explanatory Prefer absolute_path over path. Prefer max_results_count over n. Ambiguous parameters cause incorrect calls.
4
Minimal viable toolset If a human can't immediately say which tool applies in a given situation, an agent can't either. Prune overlapping tools. A bloated toolset confuses the model.
5
Token-efficient responses Tool responses consume context on every call. Return only what's needed — summarise long outputs, paginate large datasets, truncate at a sensible limit (e.g. 25,000 tokens).
6
Fail gracefully with useful errors Return structured error messages the model can act on: "File not found at /path — did you mean /other/path?" Cryptic errors cause the agent to loop or hallucinate.
Watch Out
One of the most common agentic failures is an overlapping toolset — two tools that could plausibly apply to the same situation. The agent will either pick the wrong one or alternate between them. Audit your tools: if any two have similar descriptions, merge or rename them.
Section 05

Agent Architecture Patterns

Most agentic systems combine a handful of recurring patterns. Knowing these lets you choose the right structure for your task and prompt accordingly.

When: Tasks that fit in one context window and don't require delegation. Your first choice for simple agentic tasks.

  • One model with a defined toolset, loops until task is complete or it hits a checkpoint.
  • Easiest to debug — all state is in one place.
  • Good for: code edits, file transforms, web research, building a simple static website.
When to use
Start here. Only move to multi-agent patterns when a single agent genuinely can't handle the complexity or scale of the task.

When: Tasks whose complexity is unpredictable or too large for one context window. The orchestrator breaks the task into subtasks and delegates to worker agents.

  • Orchestrator — receives the top-level goal, plans subtasks, assigns them to workers.
  • Workers — specialised agents, each with a focused toolset and narrow system prompt.
  • Workers report results back; orchestrator integrates them and decides the next step.
  • Example: a GitHub issue involves 5 files across 3 directories — orchestrator assigns one worker per file cluster.
Pro Tip
Give each worker a minimal, purpose-specific system prompt. A worker that edits CSS should not have tools or instructions for running tests. Narrow scope = fewer errors.

When: The quality of the output matters more than speed, and "good enough" on the first pass is not acceptable — writing, code review, translation quality.

  • Generator — produces the initial output.
  • Evaluator — independently scores or critiques it against defined criteria.
  • Loop repeats until the evaluator passes the output or a max iteration count is hit.
  • The evaluator can be the same model with a different system prompt, or a separate one.
Watch Out
Without a maximum iteration limit, evaluator-optimizer loops can run indefinitely. Always set a hard cap (e.g., 3–5 rounds) and define what "good enough" means explicitly in the evaluator's prompt.

When: Tasks where rushing into action before thinking causes cascading mistakes. Used in production agents like Cline (the VS Code coding agent).

  • Plan mode — agent analyses the problem and writes a step-by-step plan. No tools are called. User reviews and approves.
  • Act mode — agent executes the approved plan, one step at a time.
  • Each step waits for confirmation before proceeding to the next.
  • Dramatically reduces compounding errors on complex, multi-file tasks.
Pro Tip
Even if you don't formally implement Plan/Act modes, you can mimic this with prompting: "Before taking any action, output your full plan. Wait for my approval before proceeding."

Choosing the right pattern

PatternBest forMain risk
Single agentContained, well-scoped tasksContext window overflow
Orchestrator-WorkersLarge, parallel, or unpredictable tasksCoordination overhead
Evaluator-OptimizerQuality-sensitive outputsInfinite loops without a cap
Plan + ActComplex, multi-step, risky actionsSlower; needs user patience
Section 06

Guardrails & Human Control

The more autonomy you give an agent, the more important guardrails become. Autonomous action without oversight is how small mistakes become expensive ones.

1
Checkpoint before irreversible actions Deleting files, pushing to main, sending emails, writing to a database — any action that can't be easily undone needs an explicit pause for human review.
2
One tool call per turn (where possible) Requiring confirmation after each tool use creates a feedback loop. It prevents cascading failures and lets you catch problems before they compound.
3
Hard limits in the system prompt Be explicit: "Do not modify any file outside /project/src". "Do not call any external API not listed here." Agents will explore the full space you give them.
4
Least-privilege tools Only give the agent the tools it actually needs for this task. A read-only research agent should not have a write_file tool. Scope tools to the job.
5
Explicit escalation path Define when the agent should stop and ask rather than proceed. "If you are uncertain about the intent of a request, ask before acting." Ambiguity is the enemy of safe autonomy.
6
Dry-run mode for testing During development, prompt the agent to describe what it would do without actually doing it. Audit the plan before letting it loose on real files or APIs.
Pro Tip — Minimal footprint principle
Anthropic's guidance for safe agents: request only necessary permissions, avoid storing sensitive information beyond immediate needs, prefer reversible over irreversible actions, and err on the side of doing less and confirming when uncertain about intended scope.

Trust levels in multi-agent systems

When one agent sends instructions to another, the receiving agent has no way to verify those instructions are legitimate. Apply the same scrutiny to agent-generated instructions that you would to user input.

Prompt Injection Risk
An agent that reads external content (web pages, emails, documents) can be tricked by malicious text in that content instructing it to take unexpected actions. Always treat externally-sourced content as untrusted data, not trusted instructions.
Section 07

Worked Example: GitHub Pages Site

You want an agent to build and maintain a simple static website hosted on GitHub Pages. Here's how to apply everything in this sheet to that real task.

Step 1 — System prompt (minimal first pass)

<background_information> You are a coding agent working on a static website repository. The site uses plain HTML, CSS, and optional vanilla JavaScript. It will be hosted on GitHub Pages from the /docs folder on the main branch. File structure: /docs/index.html ← main page /docs/style.css ← stylesheet /docs/assets/ ← images and other static files </background_information> <instructions> Implement the user's requested changes to the site. Before editing any file, read it first with read_file. Make the smallest change that achieves the goal. After completing a change, summarise what you did in one sentence and list any files modified. </instructions> ## Hard limits - Only modify files inside /docs/ - Do not commit or push; only write files locally - If asked to delete a file, confirm with the user first

Step 2 — Tool definitions

Tool nameDescription (what you'd write)
read_file Read the full text content of a file at the given absolute path. Use this before any edit. Do NOT use for binary/image files.
write_file Write or overwrite a file at the given absolute path with the provided content. Always read the file first. Only use within /docs/.
list_directory List all files and directories at the given absolute path, one level deep. Use to explore the project structure before making assumptions.

Step 3 — Example user messages (what to say)

✓ Good — specific and scoped: "Add a navigation bar to index.html with links to Home, About, and Contact. Style it in style.css to match the existing colour scheme. Do not change any other part of the page." ✓ Good — with a constraint: "Update the hero section background colour to #1a1a2e. Only change the background-color property — do not refactor any other styles." ✗ Vague — avoid: "Make the site look better." ✗ Too broad — avoid: "Redesign the whole thing with a modern aesthetic."

Step 4 — Using Plan + Act for bigger changes

User message: "I want to add a blog section with an index page and three initial posts. Before doing anything, write out your full plan — what files you'll create, what changes you'll make to index.html, and in what order. Do not create any files until I approve the plan." → Agent outputs its plan. → You review, correct if needed, then say: "Approved — go ahead." → Agent executes one file at a time, reporting after each.
Pro Tip — GitHub workflow
Keep the agent working on a feature branch, not main. Instruct it explicitly: "Assume you are on a branch called feature/blog. Do not reference or modify main." Review the diff in a pull request before merging — this is your final human checkpoint.
When you're ready to go further
Once comfortable with a single-agent file-editing loop, the natural next steps are: (1) adding a run_command tool so the agent can run build scripts; (2) connecting GitHub via MCP to let the agent create PRs and read issues; (3) introducing an evaluator agent that checks HTML validity and accessibility after each change.
